# Bundling

A CEF app cannot run as a bare executable — Chromium demands a real, correctly
shaped `.app` bundle. The `cef bundle` plugin command produces one:

```sh
swift package --allow-writing-to-package-directory \
              --allow-network-connections all \
              cef bundle --product MyApp --configuration release
```

This document explains exactly what it builds and why.

## Bundle anatomy

```
MyApp.app/
└── Contents/
    ├── Info.plist
    ├── MacOS/
    │   └── MyApp                          # your product binary
    └── Frameworks/
        ├── Chromium Embedded Framework.framework/
        │   ├── Versions/A/
        │   │   ├── Chromium Embedded Framework   # the dylib we dlopen
        │   │   ├── Resources/                    # .pak files, locales, icudtl.dat
        │   │   └── Libraries/                    # libEGL, libGLESv2, vk_*,
        │   │                                     # libcef_sandbox.dylib (M138+)
        │   ├── Versions/Current -> A
        │   ├── Chromium Embedded Framework -> Versions/Current/...
        │   └── Resources -> Versions/Current/Resources
        ├── MyApp Helper.app
        ├── MyApp Helper (Alerts).app
        ├── MyApp Helper (GPU).app
        ├── MyApp Helper (Plugin).app
        └── MyApp Helper (Renderer).app
```

### Why five helpers?

Chromium runs renderers, the GPU process, plugin hosts, and notification
("alerts") services as separate processes. On macOS each process type must be
its own `.app` so it can have its own bundle identifier, Info.plist, and
(eventually) sandbox entitlements. All five contain **the same binary** — the
`cef-helper` executable — copied and renamed.

The names are **load-bearing**. CEF derives helper paths from the main
executable name, so the helper executable must be named exactly
`<main exe name> Helper (Suffix)`, and the helper `.app`s sit in the main
app's `Contents/Frameworks/`. Bundle IDs follow the same convention:

| Helper | Bundle ID suffix |
|---|---|
| `MyApp Helper.app` | `.helper` |
| `MyApp Helper (Alerts).app` | `.helper.alerts` |
| `MyApp Helper (GPU).app` | `.helper.gpu` |
| `MyApp Helper (Plugin).app` | `.helper.plugin` |
| `MyApp Helper (Renderer).app` | `.helper.renderer` |

### Versioned framework layout

CEF binary distributions ship the framework in a *flat* layout. Xcode 26's
toolchain (and modern codesign expectations) require the *versioned* bundle
layout — `Versions/A/` plus top-level symlinks. The `cef download` step
converts the framework in place, so everything downstream (bundling, signing,
Xcode projects generated by XcodeGen) sees a well-formed framework.

## Info.plist keys

Main app (generated by the plugin; override identity via `cefapp.json` next to
your target's sources, or `--bundle-id` / `--name` flags):

| Key | Value | Why |
|---|---|---|
| `LSEnvironment` | `{ MallocNanoZone = "0" }` | **Required.** Chromium's allocator is incompatible with the Nano malloc zone; without this the app can crash deep inside CEF at startup. |
| `NSPrincipalClass` | `NSApplication` | CefSwift installs its own `CEFApplication` programmatically; the plist still declares the AppKit principal class. |
| `LSMinimumSystemVersion` | `14.0` | Package floor. |
| `NSSupportsAutomaticGraphicsSwitching` | `true` | Lets dual-GPU Macs use the integrated GPU. |

Helpers:

| Key | Value | Why |
|---|---|---|
| `LSUIElement` | `1` | Helpers must never appear in the Dock or app switcher. |
| `LSFileQuarantineEnabled` | `true` | Files downloaded by renderer processes get quarantined, matching browser norms. |

## Codesigning

Signing must be **inside-out** — every nested code object signed before its
container:

1. `Chromium Embedded Framework.framework`
2. each of the five helper apps
3. the main app

The plugin performs ad-hoc signing (`codesign -s -`) in that order, which is
sufficient to run locally on the build machine.

## Distributing outside your machine

Ad-hoc-signed apps are for development. If you hand the app to someone else:

- **Quarantine:** downloaded copies carry `com.apple.quarantine`; Gatekeeper
  will refuse an ad-hoc-signed quarantined app. For ad-hoc testing the
  recipient must clear it (`xattr -dr com.apple.quarantine MyApp.app`) — or
  you sign properly:
- **Real distribution (future plugin support, doable manually today):** sign
  inside-out with a Developer ID certificate and the **hardened runtime**
  (`--options runtime`). CEF needs the standard JIT/unsigned-memory
  entitlements (`com.apple.security.cs.allow-jit`,
  `com.apple.security.cs.allow-unsigned-executable-memory`, and
  `disable-library-validation` since the CEF framework is signed by a
  different team) — apply them to the app and helpers. Then notarize
  (`xcrun notarytool submit`) and staple. Built-in hardened-runtime +
  notarization support is on the roadmap.

## Flavors

| Flavor | Contents | Use |
|---|---|---|
| `minimal` | Release framework only (~120 MB download) | Default; CI |
| `standard` | Release + Debug builds | Debugging CEF itself |

Select with `cef download --flavor standard` / `cef bundle --flavor standard`,
and pin a different CEF entirely with `--cef-version`. See
[configuration.md](configuration.md).
